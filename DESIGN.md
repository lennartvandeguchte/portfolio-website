## Problem Statement

Build a personal portfolio website that showcases projects in an authentic, creative way. The site needs to work for a mixed audience (recruiters, collaborators, clients) and grow over time as new projects are added. The differentiator is the interactive experience — specifically, kinetic typography where text itself is the interactive medium.

## What Makes This Cool

Almost no developer portfolios use typography as their primary interactive element. Everyone reaches for particles, 3D, and WebGL. This site makes *words move* — text splits apart on hover, reassembles on click, animates on scroll, morphs between states. It's an editorial magazine crossed with a motion design reel. The "playground" feeling comes from how aggressively and delightfully the typography responds to the user.

This is distinctive because:
- **It's mobile-native.** Typography animations work everywhere — no WebGL fallbacks needed.
- **It shows design taste, not just technical skill.** Getting kinetic type right requires understanding rhythm, timing, and hierarchy — harder to fake than throwing Three.js at a page.
- **It scales with content.** Each new project is a new typographic moment, and the portfolio gets more impressive as it grows.
- **It's fast.** Text is the lightest possible medium. No heavy 3D assets, no shader compilation.

## Constraints

- Must ship fast: start with 1-2 projects, add more over time
- Adding a new project should be as simple as adding a file (MDX/JSON), not rebuilding components
- Must perform well on mobile — touch interactions, reduced motion support
- Clear structure underneath (nav, sections, project pages) — "structure with soul"
- The interactive experience is the differentiator, but content must remain accessible and readable

## Premises

1. The interactive experience IS the differentiator — not the project content itself
2. Clear structure with interactive polish everywhere — "structure with soul" (not exploration-based, not a maze)
3. Mobile + performance matter as much as desktop wow-factor
4. Adding projects = adding a file, not rebuilding components

## Approaches Considered

### Approach A: "Magnetic Canvas" — Minimal Viable Wow
Single-page Next.js + Framer Motion site. Interactive particle hero, physics-based hover cards. Clean structure. Ships in ~30 min with CC.

**Pros:** Fastest to ship, mobile-friendly, dead simple content system
**Cons:** Particle hero is common, wow concentrated in hero only, could feel templated

### Approach B: "Living Gallery" — The Site as Art Installation
Full WebGL canvas via React Three Fiber with HTML overlay. Projects as 3D objects in spatial environment. Custom shaders per project.

**Pros:** Genuinely memorable, shows deep technical skill, unique per-project identity
**Cons:** WebGL mobile issues, tight performance budget, complex content authoring

### Approach C: "Kinetic Typography" — Words That Move (CHOSEN)
Typography-forward site where text is the interactive element. Large expressive type that animates on scroll, splits on hover, morphs on click. GSAP + SplitType for animations. Editorial magazine meets motion design reel.

**Pros:** Highly distinctive, inherently mobile-friendly, simple content system, feels "designed"
**Cons:** Relies on font choices + animation timing, less visual variety between projects, subtler wow

## Recommended Approach

**Approach C: Kinetic Typography** — chosen because it's the rarest approach in the portfolio space, inherently mobile-friendly (no WebGL), and shows design taste over raw technical brute force. The risk profile is low-medium, and the content system stays simple.

**Technical notes:**
- **Native scroll + ScrollTrigger:** No Lenis. GSAP ScrollTrigger works directly with native scroll. This preserves browser find-on-page, anchor links, back-button scroll restoration, and accessibility tool scroll jumping. Simpler, more reliable.
- **`useSplitType` custom hook** (highest-risk implementation component): initializes SplitType after `document.fonts.ready`, cleans up DOM on unmount, debounced re-init on window resize. Must handle React strict mode double-renders, route transitions, and fast scroll scenarios where SplitType's DOM manipulation conflicts with React's virtual DOM. Test thoroughly.
- **`useFontLoaded` custom hook**: wraps `document.fonts.ready` with a 3s timeout. Returns `{ loaded: boolean, timedOut: boolean }`. All animated components gate on this.
- **Progressive enhancement:** Server-render hero text at full opacity. On JS boot, hide text, init SplitType, then re-reveal with animations. If JS fails, user sees static but readable content. Never show a blank page.
- **Font loading:** Use `next/font` (self-hosts the font, automatic preload). SplitType init gates on `useFontLoaded`.
- **MDX approach:** `next-mdx-remote/rsc` loads MDX as data (not as page components). Frontmatter parsed at build time via `gray-matter`. Content rendered server-side.
- **Font weight animation caveat:** Animating `font-variation-settings` triggers layout recalc per frame per letter. Test on real devices. If performance is poor, fall back to opacity/scale-based effects for the weight wave. The performance budget says "CSS transforms only" — font weight is an exception that must earn its keep.
- **Accent color constraint:** `--color-accent` (#ff6b35) at 4.8:1 contrast passes AA for large text only (18px+ or 14px+ bold). NEVER use accent on `--text-small` or `--text-nav` at regular weight. For small interactive text, use `--color-text` (#f0f0f0) instead.

### Key Interactive Moments

1. **Hero Section** (height: ~85vh, first project card peeks below)
   - Your name rendered in massive type (120px+)
   - Role subtitle: "AI and software development" (or similar) — visible on load, animates in word-by-word after name (0.4s per word, 0.15s stagger)
   - **Desktop:** on hover, individual letters scatter 20-80px in random directions over 0.6s (elastic easing), reassemble on mouseout over 0.4s (power2.out). Re-hovering mid-reassembly restarts the scatter.
   - **Mobile:** auto-animate on load — letters scatter outward then reassemble once as an entrance moment (after font load + fade-in). No tap required.
   - Scroll indicator animates with kinetic text: "scroll" written in a looping animation

2. **Project Cards** (full-width, left-aligned, stacked vertically with `var(--space-lg)` gap)
   - Project title: `var(--text-h1)`, weight 400 default, left-aligned
   - Scroll-into-view: letters slide in from different directions, staggered (0.05s per letter, 0.8s total, power3.out)
   - On hover: text weight shifts 400 → 800 in a wave across the title (0.1s stagger per letter, 0.3s per letter)
   - Description: `var(--text-body)`, `var(--color-text-muted)`, reveals line-by-line with clip-path wipe (0.4s per line, 0.1s stagger)
   - Tags: `var(--text-small)`, `var(--color-text-muted)`, inline with `/` separator
   - Background image: project screenshot at 10% opacity, covers full card area, fades to 20% on hover (0.3s ease). Lazy-loaded, WebP, blurred slightly (filter: blur(2px)) to keep type readable.
   - Card border: subtle bottom border `var(--color-border)` between cards

3. **Page Transitions** (duration: 500ms total, interruptible)
   - Clicking a project: title scales up via FLIP animation (position: fixed during transition)
   - Surrounding content (other cards, nav) fades out over 200ms
   - Title moves to its target position on the project page over 300ms (power2.inOut easing)
   - Project page content fades in staggered after title lands (200ms, 50ms stagger per element)
   - Back navigation: reverse the animation (title shrinks back, cards fade in)
   - **Interruptible:** if user clicks back mid-transition, animation completes instantly to target state (no stuck states)
   - **Mobile:** simpler crossfade transition (300ms) — FLIP is complex on variable viewports
   - **Reduced motion:** instant navigation, no animation

4. **About Section**
   - Bio text that reacts to cursor proximity — words subtly shift/tilt as mouse moves near
   - Key skills or traits highlighted with color shifts on hover

5. **Contact Section**
   - Email/links rendered as large interactive type
   - Hover triggers a playful animation (bounce, stretch, or color morph)

6. **Micro-interactions Throughout**
   - Nav links with letter-stagger hover animations
   - Scroll-progress indicator using animated text
   - `prefers-reduced-motion` respected: all animations have static fallbacks

**Font loading strategy:**
- Preload variable font in `<head>` with `<link rel="preload">`
- Hide all animated text with `opacity: 0` until `document.fonts.ready` resolves
- On ready: initialize SplitType, then fade everything in together (0.3s ease)
- Fallback: if fonts haven't loaded after 3s, show content with system font (no weight animations)
- Technical note: ALL SplitType instances must initialize AFTER `document.fonts.ready`

**Reduced motion strategy (`prefers-reduced-motion: reduce`):**
- All text renders in final state instantly — no scatter, no stagger, no scroll-trigger
- Variable font weight changes on hover: instant (no transition), still visible
- Color shifts on hover: instant, no transition duration
- Page transitions: instant navigation, no morph/rearrange animation
- The site remains typographically bold and designed — it's still, not broken

**404 page:**
- Large kinetic "404" with subtle letter animation
- "This project wandered off" in the role subtitle style
- Link back to homepage

**Desktop (> 1024px):**

### Accessibility

**Keyboard navigation:**
- All interactive elements focusable with Tab
- Project cards: Enter to open, visible focus ring (2px solid var(--color-accent), 2px offset)
- Skip-to-content link (visually hidden, appears on Tab)
- Page transitions: triggered by Enter/Space on focused card
- Escape: close/back from project page

**Screen readers:**
- Semantic HTML: `<main>`, `<nav>`, `<article>` for project case studies, `<section>` with `aria-label` for each homepage section
- Hero animation: `aria-hidden="true"` on the animated letter spans, readable name in a visually hidden `<h1>`
- Project cards: `<a>` with descriptive text, not div-with-onclick
- Animation states: no aria-live regions (animations are decorative, not content)

**Visual accessibility:**
- Contrast ratios: #f0f0f0 on #0a0a0a = 18.1:1 (AAA). #888888 on #0a0a0a = 5.3:1 (AA). #ff6b35 on #0a0a0a = 4.8:1 (AA for large text only — accent used only on large interactive text and hover states, never for small body text)
- Touch targets: minimum 44x44px on all interactive elements
- Focus visible: always shown (no `:focus` without `:focus-visible`)
- `prefers-reduced-motion`: fully designed (see Interaction States above)
- `prefers-color-scheme`: dark-only for v1 (the dark palette IS the design)

### Performance Budget
- All text animations: CSS transforms only (GPU-accelerated, no layout thrashing)
- Font: single variable font file, preloaded, WOFF2 (~50-80KB)
- No images on homepage unless project thumbnails (lazy-loaded, WebP, max 200KB each)
- Target: Lighthouse performance >90 mobile, <2s FCP, <100ms INP
- No horizontal scroll, no WebGL, no canvas
